## 🌿 브랜치 규칙

이름은 **영어 이름 또는 영문 표기만** 사용해주세요.  
예: `juhwan-kim`, `kim-juhwan`  

각자 자신의 브랜치에서만 작업하며,  
`main` 브랜치에는 **직접 push하지 않습니다.**

---

### 🔤 일반적인 브랜치 이름 규칙

- **소문자 사용:** 브랜치 이름은 모두 소문자로 작성합니다.  
- **하이픈(-) 사용:** 단어와 단어 사이는 하이픈으로 구분합니다.  
- **간결성:** 이름은 짧고 명확하게 작성합니다.  
- **의미 전달:** 브랜치의 기능이나 목적을 알 수 있도록 이름을 짓습니다.  

예시:  
`feature-login`, `fix-bug-image-upload`, `juhwan-kim`

---

## 💾 커밋 규칙

- **하루 학습 분량 단위로 commit** 해주세요.  
- 하루에 진행하는 양이 많을 경우, 여러 번 commit 하셔도 좋습니다!  

예시:  
```bash
git commit -m "Day 03 - 익스프레스"
git commit -m "Day 03 - 노드의 모듈"
```

## 📋 일반적인 git 커밋 원칙
- Header, Body, Footer는 빈 행으로 구분합니다.
- 제목에는 마침표를 붙이지 않으며, 50글자 이내로 작성하는 것이 좋습니다.

```bash
타입(스코프): 제목 // Header - 스코프의 경우 생략 가능

본문 // Body - 상세한 내용 설명(마찬가지로 해더 내용이 충분하다면 생략 가능)

바닥글 // Footer - 참조 정보, 이슈 요약 (생략가능)
```
[스코프 종류]
- feat:	새로운 기능에 대한 커밋
- fix:	버그 수정에 대한 커밋
- build:	빌드 관련 파일 수정 / 모듈 설치 또는 삭제에 대한 커밋
- chore:	그 외 자잘한 수정에 대한 커밋
- ci:	ci(Continuous Integration, 코드의 변경점을 기존 소스 코드에 통합하는 것) 관련 설정 수정에 대한 커밋
- docs:	문서 수정에 대한 커밋
- style:	코드 스타일 혹은 포맷 등에 관한 커밋
- refactor:	코드 리팩토링에 대한 커밋
- test:	테스트 코드 수정에 대한 커밋
- perf:	성능 개선에 대한 커밋

예시:  
```bash
git commit -m "refactor(auth): useAuth 커스텀 훅 도입으로 로그인 로직 개선

기존에 Login, Signup 컴포넌트 등 여러 곳에 흩어져 있던
인증 관련 상태(user, token) 및 API 호출 로직을
useAuth라는 단일 커스텀 훅으로 통합합니다.

- 컴포넌트에서 비즈니스 로직 분리
- 인증 상태 관리 로직 중앙화 및 재사용성 증대
- 기능적 변경(버그 수정이나 새 기능)은 포함하지 않음"
```
